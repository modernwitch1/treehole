# 浙工商树洞邮件服务商注册与配置手册

更新日期：2026-07-13

本文只针对注册验证码和密码重置这类事务邮件。不要把事务邮件配置成营销群发，也不要通过注册多个账号轮换额度。项目内置的备用链只在网络超时、连接失败或服务商 5xx 故障时切换；遇到 400、401、403、429、发件人未验证或额度不足，会停止发送并保留明确的故障原因。

推荐顺序：Brevo 主服务商，Mailjet 备用，SMTP2GO 再备用，保留 Resend 作为最后的已配置备用。生产环境不需要同时开通全部服务，至少先开通 Brevo 和一个备用服务并完成发件域验证。

## 二、注册和域名验证

### 1. 先准备发送子域名

建议使用独立子域名，例如 `notify.unidating.top`，发件地址使用：

```text
no-reply@notify.unidating.top
```

这样可以把通知邮件的信誉和主站域名的其他用途分开。也可以直接使用 `no-reply@unidating.top`，但必须保证所有服务商的域名验证记录都正确。

在 DNS 服务商中按每个平台控制台给出的记录添加：

- SPF：整个域名只能保留一条 SPF TXT 记录。如果多个服务商都要求 SPF `include`，把它们合并到同一条记录，不能创建多条 SPF。
- DKIM：每个服务商通常使用不同 selector，可以并存。记录名称、值和 TTL 以控制台显示为准。
- DMARC：整个域名只保留一条 DMARC TXT 记录。建议先使用 `p=none` 观察投递，再逐步收紧策略。

不要照抄其他服务商的 DNS 示例；每个账户生成的 DKIM 和 SPF 值都不同。

### 2. 注册 Brevo（推荐主服务商）

1. 打开 [Brevo 注册页](https://www.brevo.com/)，创建账号并完成邮箱验证。
2. 在设置中找到发件人/发送域，添加 `notify.unidating.top` 并完成 DNS 验证。
3. 在 Transactional/SMTP 设置中确认账户已允许发送事务邮件。
4. 在 SMTP & API 设置中创建 API key，权限只用于发送邮件；复制后立即保存，平台通常不会再次展示完整密钥。
5. 记下 API key，稍后填入 `BREVO_API_KEY`。

### 3. 注册 Mailjet（推荐备用）

1. 打开 [Mailjet 免费套餐](https://www.mailjet.com/pricing/)，选择 Free 并完成注册。
2. 在 Sender domains & addresses 中添加并验证 `notify.unidating.top`。
3. 在 API/SMTP 设置中创建 API key 和 Secret key。
4. 分别保存到 `MAILJET_API_KEY` 和 `MAILJET_API_SECRET`。

### 4. 注册 SMTP2GO（第二备用）

1. 打开 [SMTP2GO 免费套餐](https://www.smtp2go.com/pricing/)，选择 Free。
2. 在 Sending → Verified Senders 中验证发送域。验证域比只验证单个地址更适合生产环境。
3. 在 Sending → API Keys 创建 API key，只授予发送权限。
4. 保存到 `SMTP2GO_API_KEY`。域名验证完成后，25 封/小时的临时限制会解除。

### 5. 注册 Mailgun（可选）

1. 打开 [Mailgun 免费套餐](https://www.mailgun.com/pricing/)，创建账号。
2. 添加并验证 `notify.unidating.top`，按控制台提示配置 SPF、DKIM 和跟踪记录。
3. 创建发送 API key，保存到 `MAILGUN_API_KEY`。
4. 把域名保存到 `MAILGUN_DOMAIN`，例如 `notify.unidating.top`。
5. 美国区域使用 `https://api.mailgun.net`；欧盟区域请将 `MAILGUN_API_BASE_URL` 改成 `https://api.eu.mailgun.net`。

## 三、项目配置

在服务器上的 `apps/api/.env` 中填写真实值。不要把密钥写入 Git、前端环境变量、截图或聊天记录。

```dotenv
NODE_ENV=production

# 主服务商；MAIL_PROVIDER_ORDER 决定故障转移顺序
MAIL_DRIVER=brevo
MAIL_PROVIDER_ORDER=brevo,mailjet,smtp2go,resend
MAIL_FAILOVER_ON_TRANSIENT=true
MAIL_PROVIDER_TIMEOUT_MS=8000
MAIL_PROVIDER_FAILURE_THRESHOLD=3
MAIL_PROVIDER_COOLDOWN_SECONDS=120

MAIL_FROM=no-reply@notify.unidating.top
MAIL_FROM_NAME=浙工商树洞

BREVO_API_KEY=在Brevo复制的密钥
BREVO_API_URL=https://api.brevo.com/v3/smtp/email

MAILJET_API_KEY=Mailjet公钥
MAILJET_API_SECRET=Mailjet私钥
MAILJET_API_URL=https://api.mailjet.com/v3.1/send

SMTP2GO_API_KEY=SMTP2GO密钥
SMTP2GO_API_URL=https://api.smtp2go.com/v3/email/send

# 如果启用 Mailgun，再填写这三项
MAILGUN_API_KEY=
MAILGUN_DOMAIN=
MAILGUN_API_BASE_URL=https://api.mailgun.net

# 若保留 Resend 作为最后备用
RESEND_API_KEY=
```

本地开发继续使用 maildev，不要把生产 API key 填入本地 `.env`：

```dotenv
MAIL_DRIVER=smtp
MAIL_PROVIDER_ORDER=
MAIL_FROM=no-reply@forum.local
SMTP_HOST=localhost
SMTP_PORT=1025
```

## 四、部署和验证

在服务器项目目录执行：

```bash
pnpm --filter @forum/api build
pm2 reload forum-api
pm2 save
```

然后在注册页面完成一次真实测试。检查点：

1. 首选服务商的控制台能看到一封事务邮件。
2. 收件人收到验证码，验证码与页面注册记录一致。
3. API 日志只显示脱敏邮箱和服务商名称，不显示 API key。
4. 模拟主服务商暂时不可用时，备用服务商能接收邮件；不要通过打满额度来测试切换。
5. 邮件服务商控制台中的 SPF、DKIM、DMARC 均通过，且没有明显退信或投诉。

## 五、故障判断

| 日志现象                     | 含义                         | 处理方式                                                  |
| ---------------------------- | ---------------------------- | --------------------------------------------------------- |
| `provider failed (http/401)` | API key 无效或权限不足       | 重新创建最小权限发送 key，并更新服务器环境变量            |
| `provider failed (http/403)` | 账户、域名或发件人未获准发送 | 完成账户审核和发件域验证，确保 `MAIL_FROM` 与已验证域一致 |
| `provider failed (http/429)` | 触发限流或达到配额           | 等待额度恢复或升级套餐；代码不会切换服务商规避该限制      |
| `provider failed (http/5xx)` | 服务商临时故障               | 代码会尝试下一个已配置服务商，并写入 Redis 熔断状态       |
| `circuit-open`               | 某服务商连续故障，暂时被隔离 | 等待冷却时间后自动恢复；也要查看服务商状态页              |
| `没有配置可用的邮件服务商`   | 所有候选项缺少必要密钥或配置 | 检查 `MAIL_PROVIDER_ORDER` 和对应 API key                 |
| 邮件进入垃圾箱               | 域名信誉或认证不完整         | 检查 SPF/DKIM/DMARC、From 域名一致性、退信和投诉率        |

服务商顺序是可靠性配置，不是配额叠加器。真正需要稳定发送几百封以上的注册通知时，应使用一个经过验证的付费事务邮件套餐，或者向服务商申请提高限额；这样比多个免费账号互相切换更稳定，也不会让域名信誉变得不可控。
