# 浙工商树洞美食模块：管理员管理手册

版本：2026-07

## 1. 系统边界

美食模块把学生端内容和商家运营端分开：学生可以浏览食堂、窗口、产品、商家动态和匿名评价；商家员工只能进入商家后台，不能查看论坛帖子、私信、聊天室或其他非美食数据。

生产入口：

- 管理员后台：`https://admin.unidating.top`
- 商家后台：`https://merchant.unidating.top`
- 学生端美食模块：论坛主站中的“美食”入口

商家后台使用独立的 `FoodStaffAccount`、商家会话和商家权限关系。不要把员工账号当作论坛账号发放，也不要把邀请链接发送到公开群组。

## 2. 角色与权限

| 角色        | 可做的事                                                                                 | 不可做的事                                   |
| ----------- | ---------------------------------------------------------------------------------------- | -------------------------------------------- |
| 超级管理员  | 全部商家、食堂、窗口、员工、邀请、产品、动态、评价和回复管理；可以使用商家后台的平台视图 | 不应绕过审计或共享账号                       |
| 管理员      | 管理商家生命周期、食堂和窗口，邀请/撤销员工，审核美食内容                                | 不能查看论坛非美食内容（按后台既有权限执行） |
| 版主        | 按现有后台权限处理允许范围内的内容审核                                                   | 不能管理商家、员工和邀请                     |
| 店主 owner  | 商家后台维护店铺资料、窗口、产品、动态、评价回复                                         | 不能管理平台其他商家或平台管理员             |
| 编辑 editor | 维护本店窗口、产品、动态和回复评价                                                       | 不能修改店铺核心资料；不能管理员工           |
| 只读 viewer | 查看本店产品、动态和评价                                                                 | 不能写入、回复、启停窗口或修改资料           |

后端会再次校验角色，前端隐藏按钮不是安全边界。商家员工的所有查询都按当前账号的商家关系过滤；平台管理员在商家后台可切换所有“启用”商家，但仍不能看到论坛模块数据。

## 3. 标准开店流程

### 3.1 创建食堂

进入“美食模块”，在“新增食堂”填写：

1. 英文标识，例如 `xingyun`、`liushui`；只能使用小写字母、数字和连字符。
2. 展示名称，例如“行云苑”“流水苑”。
3. 如有需要，再补充说明。

创建后默认启用。停用食堂不会删除窗口、产品或历史评价，只会停止学生端展示。

### 3.2 创建商家

在“新增商家”填写唯一英文标识和展示名称。管理员创建的商家默认进入“已启用”状态；如果尚未准备好运营资料，创建后先切换为“待启用”或“已暂停”。

商家状态含义：

- `待启用`：资料尚未完成，商家员工不能正常进入有效店铺。
- `已启用`：学生端可展示符合发布条件的内容，商家员工可进入后台。
- `已暂停`：紧急下线商家，员工会因没有有效店铺权限而无法继续操作。
- `已关闭`：结束合作或永久停止运营，保留历史数据用于审计。

状态变更会写入审计日志。暂停/关闭前应先确认是否需要保留公开证据、处理未完成评价和通知商家。

### 3.3 配置窗口

在“新增窗口”选择商家、食堂并填写窗口名称；楼层默认为二楼，可在窗口管理中修正。建议统一填写窗口编号和位置说明，例如“二楼东侧 2-08”。

窗口停用只影响新内容投放和学生端展示，历史评价不删除。重新启用前确认对应商家和食堂仍处于启用状态。

## 4. 员工邀请与账号生命周期

### 4.1 发放邀请

在“邀请商家后台员工”中选择商家、填写工作邮箱并选择角色，然后点击“生成邀请”。系统生成一次性邀请链接，有效期 72 小时。

操作要求：

1. 通过已核验的工作邮箱或一对一渠道发送链接。
2. 不要把链接放进公开群、论坛帖子或截图中。
3. 链接只显示一次；如遗失或泄露，立即撤销并重新生成。
4. 同一个商家和邮箱重新邀请时，旧的待处理邀请会失效。

### 4.2 查看和撤销

“商家员工权限”显示账号、邮箱、所属商家、角色和状态。点击“撤销”后：

- 员工关系变为 `revoked`；
- 该账号的现有商家会话立即撤销；
- 后端后续每次请求都会重新检查有效商家关系。

员工在职时可以直接调整 `店主 / 编辑 / 只读` 角色；已撤销员工可以在商家仍处于启用状态时恢复。恢复会清除员工关系的撤销时间，但不会恢复已经过期的邀请，也不会恢复已经被撤销的历史会话，员工需要重新登录。

“待处理邀请”只显示尚未过期的邀请。过期邀请会被标记为 `expired`，不能再接受；需要重新生成邀请。

不要通过数据库手工修改账号状态来替代后台操作；紧急处置时至少在工单中记录商家、账号、原因和执行时间。

## 5. 美食内容审核

### 5.1 审核状态

审核队列支持：

- `待审核`：商家新建或编辑后的内容，尚未公开；
- `已通过`：允许学生端展示；
- `已隐藏`：保留内容但不在学生端展示；
- `已删除`：作为历史状态保留，通常只由更高权限流程处理。

商家编辑产品或动态的名称、介绍、正文后，会重新进入 `待审核`。仅切换在售状态、窗口运营状态等不改变文案的操作不会重复创建文案审核。

### 5.2 队列操作

在“内容审核队列”中：

1. 先按状态筛选；
2. 使用关键词搜索标题、产品名或评价正文；
3. 查看商家、窗口和匿名标识；
4. 单条点击“通过/隐藏”，或勾选同类内容后批量处理；
5. 使用分页切换，不要依赖浏览器一次加载全部历史数据。

批量操作仍按内容逐条提交，页面会显示成功数/总数。出现部分失败时先刷新队列，确认剩余状态后再重试。每一条处置都会记录操作类型、目标和备注（如有）。

### 5.3 审核重点

- 产品名称、价格和窗口归属是否一致；
- 广告内容是否清楚标注优惠条件、时间和适用窗口；
- 是否包含无法验证的承诺、外部收款方式或与论坛无关的营销；
- 商家回复是否针对评价，不泄露学生身份或订单信息；
- 定时发布内容的发布时间不得晚于失效时间；
- 发现疑似隐私、诈骗、骚扰或违法内容时先隐藏，再按平台安全流程升级。

## 6. 运营数据与审计

页面顶部指标用于排班和待办排序：启用商家、待处理内容、活跃员工、待接受邀请、启用食堂。指标是实时查询结果，不是财务报表。

重点审计动作包括：

- 创建/更新食堂、商家、窗口；
- 商家状态改变；
- 员工邀请、撤销和权限变更；
- 产品、动态、评价、官方回复的通过、隐藏、恢复；
- 超级管理员进入平台商家视图。

审计记录用于追责和故障排查，不要把邮箱、邀请 token 或密码写入备注。邀请 token 只通过响应返回给生成者，数据库保存的是哈希。

## 7. 故障处理手册

### 员工无法登录

1. 确认登录入口是 `merchant.unidating.top`。
2. 确认使用邀请邮箱和员工自行设置的密码；超级管理员可使用论坛用户名或邮箱及论坛密码。
3. 检查邀请是否过期、是否已经接受、商家是否启用。
4. 检查员工关系是否被撤销。
5. 若浏览器提示会话过期，刷新页面；系统会尝试用安全刷新会话恢复登录。

不要要求员工把密码发给管理员。管理员应撤销旧账号/邀请并重新发放。

### 内容通过后学生端看不到

检查顺序：商家状态、食堂状态、窗口状态、产品是否 `isAvailable`、内容是否在定时发布时间之后且未超过失效时间、页面是否仍处于缓存状态。必要时使用学生端匿名窗口复核。

### 错误地隐藏了内容

在审核队列切换到“已隐藏”，确认目标后使用“恢复/通过”操作；如果内容是商家重新编辑产生的审核版本，恢复前应重新阅读当前版本，避免把旧版本误放行。

### 需要停业

先保留必要证据和审计信息，再将商家切换为“已暂停/已关闭”。不要删除商家、窗口、产品和评价记录；如需永久清理，必须另走数据保留和合规审批流程。

## 8. 管理员日常检查清单

- 每日查看待审核总数和过期邀请；
- 新商家先配置食堂、窗口，再邀请员工；
- 员工离职或合作终止后立即撤销权限；
- 高风险广告、评价和回复先隐藏再升级；
- 每周抽查商家后台访问关系、停用窗口和长期未处理邀请；
- 不共享超级管理员密码，不在聊天工具中转发邀请 token；
- 重大状态变更在内部工单记录原因、执行人和复核人。
