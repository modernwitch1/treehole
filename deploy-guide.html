2. EC2 服务器规格怎么选

最低配置（开发 / 演示）

配置项 推荐 说明
实例类型 t3.medium 2 vCPU, 4 GB 内存
存储 20 GB gp3 EBS 通用型 SSD
系统 Ubuntu 22.04 LTS 稳定, 社区支持好
带宽 按量付费 (100 Mbps) AWS 默认

推荐配置（正式上线）

配置项 推荐 说明
实例类型 t3.large 或 c7g.large 2-4 vCPU, 8 GB 内存
存储 50 GB gp3 足够存放数据库 + 注册截图附件
系统 Ubuntu 22.04 LTS —
带宽 按量付费 —

选择建议

  • t3.medium（2C4G）：适合初期测试。4G 跑 API + Web + Admin + PostgreSQL + Redis 刚好够用
  • t3.large（2C8G）：适合正式上线。内存充裕，不会轻易 OOM
  • c7g.large（4C8G）：AWS Graviton 芯片，性价比更高。如果预算允许直接选这个
  • 存储：20 GB 仅够系统+代码。建议 至少 30 GB，数据库和注册截图会随时间增长
  • 省钱提示：如果只是测试，可以用 t3.micro（免费套餐 1C1G），但无法同时跑三个服务
预算参考（美东区域，2025 年）
t3.medium: ~$30/月 · t3.large: ~$60/月 · c7g.large: ~$50/月 · 存储 30 GB gp3: ~$3/月

4. 创建 EC2 实例

4.1 登录 AWS 控制台

打开 console.aws.amazon.com，搜索 "EC2" 进入 EC2 控制台。

4.2 启动实例（关键步骤）

  1. 点击 "Launch instance"
  2. Name：填入 zjgsu-forum
  3. Application and OS Images：选择 Ubuntu Server 22.04 LTS (HVM), SSD Volume Type，架构选 x86
  4. Instance type：根据上节建议选择（起步选 t3.medium）
  5. Key pair (login)：点击 "Create new key pair"
    — 类型选 RSA，格式选 .pem（macOS/Linux）
    — 下载后保存到 安全的地方（例如 ~/.ssh/zjgsu-forum.pem）
    — ⚠ 这个文件丢了就再也连不上服务器了！
  6. Network settings：
    ✅ Allow SSH traffic from → Anywhere
    ✅ Allow HTTP traffic from the internet
    ✅ Allow HTTPS traffic from the internet
  7. Configure storage：设置 30 GB，类型 gp3
  8. 点击 "Launch instance"

4.3 设置安全组（防火墙）

实例启动后，需要额外开放几个端口（如果测试阶段直接暴露端口用）：

  1. 左侧菜单 → Security Groups → 找到你的安全组 → Edit inbound rules
类型 协议 端口 来源 说明
SSH TCP 22 0.0.0.0/0 远程连接
HTTP TCP 80 0.0.0.0/0 网站 HTTP
HTTPS TCP 443 0.0.0.0/0 网站 HTTPS
自定义 TCP TCP 3001 0.0.0.0/0 Web 前端（测试用）
自定义 TCP TCP 3002 0.0.0.0/0 Admin 后台（测试用）
🚨 安全提醒
端口 3001/3002 只有测试阶段才直接开放。正式上线后通过 Nginx 反代 + 域名访问，然后删除这两条规则。
PostgreSQL（5432）和 Redis（6379）永远不要开放给公网。

4.4 获取公网 IP + 弹性 IP

  • 实例详情页可看到 Public IPv4 address（例如 54.123.45.67），记住它
  • 左侧菜单 → Elastic IPs → Allocate → Associate 到你的实例
  • 弹性 IP 绑定到运行中的实例不收费，重启不会丢 IP

7. 安装 PostgreSQL & Redis

7.1 安装 PostgreSQL 16

# 添加 PostgreSQL 官方仓库
sudo sh -c 'echo "deb http://apt.postgresql.org/pub/repos/apt $(lsb_release -cs)-pgdg main" > /etc/apt/sources.list.d/pgdg.list'
curl -fsSL https://www.postgresql.org/media/keys/ACCC4CF8.asc | sudo gpg --dearmor -o /etc/apt/trusted.gpg.d/postgresql.gpg
sudo apt update
sudo apt install -y postgresql-16 postgresql-client-16

# 启动
sudo systemctl enable postgresql
sudo systemctl start postgresql

# 创建数据库 + 用户
sudo -u postgres psql -c "CREATE USER forum WITH PASSWORD 'your_prod_password_here';"
sudo -u postgres psql -c "CREATE DATABASE forum OWNER forum;"
sudo -u postgres psql -c "GRANT ALL PRIVILEGES ON DATABASE forum TO forum;"
sudo -u postgres psql -d forum -c "CREATE EXTENSION IF NOT EXISTS citext;"
sudo -u postgres psql -d forum -c "CREATE EXTENSION IF NOT EXISTS ltree;"
sudo -u postgres psql -d forum -c "CREATE EXTENSION IF NOT EXISTS pgcrypto;"
🔒 不要用 forum_dev_password 作为生产密码！
上面 your_prod_password_here 请替换为强随机密码。

7.2 安装 Redis 7

# 安装
sudo apt install -y redis-server

# 设置 supervised systemd
sudo sed -i 's/^supervised no/supervised systemd/' /etc/redis/redis.conf

# 启动
sudo systemctl enable redis-server
sudo systemctl start redis-server

# 验证

11. Nginx 反向代理 + 域名

11.1 安装 Nginx

sudo apt install -y nginx
sudo systemctl enable nginx

11.2 配置站点

sudo nano /etc/nginx/sites-available/forum

把下面配置中的域名改成你自己的（unidating.top, manage.unidating.top）。管理后台用一个不易猜到的子域名： 

# ===== 论坛前端 =====
server {
    listen 80;
    server_name unidating.top www.unidating.top;

    # 上传文件大小限制（截图上传）
    client_max_body_size 10M;

    location / {
        proxy_pass http://127.0.0.1:3001;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_cache_bypass $http_upgrade;
    }
}

# ===== 管理后台（隐藏入口，用不易猜到的子域名）=====
server {
    listen 80;
    server_name manage.unidating.top;

    location / {
        proxy_pass http://127.0.0.1:3002;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_cache_bypass $http_upgrade;
    }
}

11.3 启用 + 重启

sudo ln -s /etc/nginx/sites-available/forum /etc/nginx/sites-enabled/
sudo rm -f /etc/nginx/sites-enabled/default     # 删除默认站点
sudo nginx -t                                     # 测试配置
sudo systemctl reload nginx                       # 应用

11.4 先用 IP 直连验证部署

在配域名之前，先用 IP 确认服务能跑通：

  • http://你的弹性IP:3001 → 应看到论坛首页

11-2. 阿里云域名 DNS 配置（unidating.top → AWS EC2）

你的情况
域名 unidating.top 是在阿里云购买的，但服务器在 AWS EC2 上。 DNS 的作用就是告诉浏览器：「访问 unidating.top 时，请去 AWS 那个 IP 地址找服务器」。 操作很简单：在阿里云后台添加几条记录即可。

第 1 步：登录阿里云 DNS 控制台

  1. 打开 dns.console.aliyun.com
  2. 在域名列表里找到 unidating.top
  3. 点击右侧的 "解析设置"

第 2 步：添加 DNS 记录

点击 "添加记录"，依次添加以下 3 条：

记录类型 主机记录 记录值 TTL 说明
A @ 你的 AWS 弹性 IP（如 54.123.45.67） 600 主域名 unidating.top 指向服务器
A manage 你的 AWS 弹性 IP 600 manage.unidating.top 指向管理后台（隐藏入口）
A www 你的 AWS 弹性 IP 600 www.unidating.top 也指向论坛
📝 字段解释（新手必看）
主机记录 @ = 代表域名本身（unidating.top）
主机记录 manage = 代表子域名 manage.unidating.top
记录值 = 你的 AWS EC2 弹性 IP 地址
TTL 600 = DNS 缓存 10 分钟，生效快

第 3 步：验证 DNS 是否生效

# 在本地电脑执行（替换为你的 IP）
nslookup unidating.top
# 应返回你设置的 AWS IP 地址

# 或直接用 ping
ping unidating.top
# 应返回你的 AWS IP
⏳ DNS 生效需要时间
阿里云修改 DNS 后一般 5-15 分钟内生效。如果 nslookup 返回的不是你的 IP，请耐心等一会再试。
如果超过 1 小时还没生效，检查阿里云控制台里域名状态是否是「正常」。

第 4 步：更新 Nginx 配置使用你的域名

回到第 11 节的 Nginx 配置，把所有 forum.yourdomain.com 替换为 unidating.top，管理后台用 manage.unidating.top： 

sudo nano /etc/nginx/sites-available/forum
# 论坛前端
server {
    listen 80;
    server_name unidating.top www.unidating.top;
    client_max_body_size 10M;
    location / {
        proxy_pass http://127.0.0.1:3001;
        ...
    }
}

# 管理后台
server {
    listen 80;
    server_name manage.unidating.top;
    location / {
        proxy_pass http://127.0.0.1:3002;
        ...
    }
}
# 重启 Nginx
sudo nginx -t && sudo systemctl reload nginx

Certbot 安装后会自动添加定时任务（systemd timer），证书到期前自动续。你可以安心不管它。
手动续期命令：sudo certbot renew

13. 配置邮箱验证码发送（AWS SES / 第三方 SMTP）

为什么要配邮箱？
注册时选择「邮箱验证」方式，后端会生成 6 位随机验证码存到数据库，然后 发邮件 给用户。 如果你不配邮箱发送，验证码虽然生成了但用户收不到。
mock 模式下验证码固定为 123456，不依赖真实邮件。生产环境需要按下面步骤配。

方案 A：AWS SES（推荐，和 EC2 在同一生态）

13A.1 什么是 AWS SES

SES（Simple Email Service）是 AWS 的邮件发送服务。EC2 上的应用通过 SES 发邮件免费用额 每天 62,000 封（从 EC2 发送），足够校园论坛使用。

13A.2 申请 SES 生产访问

  1. 打开 AWS SES 控制台
  2. 右上角区域选择 Asia Pacific (Tokyo) / ap-northeast-1
  3. 先完成发件身份验证和生产访问申请
  4. 推荐给 EC2 绑定 IAM Role，让后端通过 SES API 发信，不在服务器保存长期 access key

13A.3 验证发件邮箱地址

SES 默认在 sandbox（沙盒） 模式下，只能向已验证的邮箱发信。你需要：

  1. SES 控制台 → 左侧 "Verified identities" → "Create identity"
  2. 选 "Email address"，填入你要用来发验证码的邮箱（比如 noreply@unidating.top）
  3. 点 "Create identity"
  4. AWS 会向这个邮箱发一封确认邮件 → 打开邮件 → 点确认链接
  5. 状态变成 Verified 就 OK 了
  6. 同样操作，把你自己的常用邮箱也验证一下（加为测试收件人）

13A.4 升级到生产模式（可选，公开上线时做）

沙盒模式下只能发给你验证过的邮箱。正式上线后需要：

  1. SES 控制台 → 左侧 "Account dashboard"
  2. 点击 "Request production access"
  3. 填写申请（说明你是校园论坛，仅用于发送邮箱验证码和通知邮件）
  4. AWS 审核通过后（通常 1-2 天），就可以向任意邮箱发送了
沙盒模式足够初期测试
你注册一个测试邮箱，在 SES 里把自己和 noreply@unidating.top 都验证了。然后注册时用那个测试邮箱就能收到验证码。
等正式上线时再去申请生产模式。

13A.5 配置项目 .env

项目后端使用 AWS SES API 发送验证码。给 EC2 绑定有 SES 发信权限的 IAM Role 后，编辑 API 的 .env： 

nano ~/forum/apps/api/.env

修改邮件相关配置：

MAIL_DRIVER=ses
AWS_REGION=ap-northeast-1
MAIL_FROM=你在SES东京区域验证过的发件邮箱
SES SMTP 服务器地址由区域决定
如果你的 EC2 在美东（us-east-1）：email-smtp.us-east-1.amazonaws.com
如果在东京（ap-northeast-1）：email-smtp.ap-northeast-1.amazonaws.com
在 SES 控制台 "SMTP settings" 页面能看到完整的服务器地址。

方案 B：用第三方 SMTP（阿里云企业邮箱 / QQ邮箱 / 163邮箱）

如果你不想折腾 AWS SES，也可以用任何 SMTP 邮箱发送。以 163 邮箱 为例：

  1. 登录 mail.163.com → 设置 → POP3/SMTP/IMAP → 开启 SMTP 服务
  2. 设置一个「授权码」（不是登录密码，是专门的客户端密码）
  3. .env 配置：
MAIL_DRIVER=smtp
MAIL_FROM=你的163邮箱@163.com
SMTP_HOST=smtp.163.com
SMTP_PORT=465
SMTP_USER=你的163邮箱@163.com
SMTP_PASSWORD=你的163授权码
第三方邮箱的每日发送限制
163 邮箱每天大概能发 50-100 封，QQ 邮箱类似。如果用户量大，建议还是用 AWS SES（每天 62,000 封免费）。

13.5 验证邮件发送是否成功

配完邮箱后重启 API：

pm2 restart forum-api
pm2 logs forum-api  # 看有没有报错

然后在注册页用邮箱方式注册，验证码应该会发到对应邮箱（实际代码会在 API 日志里打印验证码内容）。

16. 常见问题

Q: 注册后验证邮件收不到

mock 模式下不需要真实邮件，验证码固定为 123456。
生产模式需要配置 AWS SES（Simple Email Service）账户并激活发件邮箱，然后修改 .env 里的 SMTP 配置。

Q: 截图上传失败 / 管理员看不到截图

mock 模式下上传返回的是一个占位 URL，仅用于测试流程。生产模式需要配置 S3 存储桶。
注册截图会作为 registration_requests.screenshot_url 存到数据库。

Q: 部署后网页空白 / 502 Bad Gateway

pm2 status 确认三个进程都是 online。
pm2 logs forum-web 查看错误日志。
sudo systemctl status nginx 确认 Nginx 在运行。

Q: 数据库连接失败（API 日志显示 Prisma connection error）

检查 .env 里 DATABASE_URL 的用户名和密码。
sudo systemctl status postgresql 确认 PostgreSQL 运行中。
psql -U forum -d forum -h localhost -W 手动测试。

Q: 服务器内存不够

# 查看内存
free -h

# 如果接近 100%，优先级:
# 1. 先停掉管理后台
pm2 stop forum-admin

# 2. 升级实例（AWS 控制台 → 停止实例 → 更改实例类型 → 启动）
# 3. 临时加 2GB Swap
sudo fallocate -l 2G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile
echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab

Q: 如何更新代码

# 拉取最新
cd ~/forum && git pull
pnpm install --frozen-lockfile

# 重新生成 Prisma（如果 schema 有变）
pnpm --filter @forum/api exec prisma generate

# 执行新迁移（如有）
pnpm --filter @forum/api exec prisma migrate deploy

# 重新构建
pnpm --filter @forum/api build
NEXT_PUBLIC_USE_MOCK=true pnpm --filter @forum/web build
NEXT_PUBLIC_USE_MOCK=true pnpm --filter @forum/admin build

# 重启
pm2 restart all

Q: 什么时候把 mock 关掉？

等后端的帖子/评论/私信等业务逻辑全部开发完成后（API Slice 2-10）。
届时将 .env.local 里的 NEXT_PUBLIC_USE_MOCK 改为 false，重新构建并重启即可。
注册/审批功能前后端已经完整对接，mock 只影响帖子列表和评论等板块。
